# OSINT backend status

Live health check for the 16 OSINT backends registered in
[`scripts/osint-status.sh`](https://github.com/Opita-Code/dark-research-mcp/blob/main/scripts/osint-status.sh).

Updated weekly by the
[`osint-status`](https://github.com/Opita-Code/dark-research-mcp/actions/workflows/osint-status.yml)
workflow and on every PR by the
[`go-test`](https://github.com/Opita-Code/dark-research-mcp/actions/workflows/go-test.yml)
`osint-smoke` job.

The canonical report lives at
[`BACKEND_STATUS.md`](https://github.com/Opita-Code/dark-research-mcp/blob/main/BACKEND_STATUS.md)
in the repo. This page is a Jekyll-rendered copy that the workflow
regenerates every Monday at 06:00 UTC and every time
`osint-smoke` produces a new report on a PR.

## How the probe works

The probe sends a single lightweight GET to each backend with a
10-second timeout, classifies the response (`ok`, `redirect`,
`auth_required`, `timeout`, `fail`), and writes a markdown table
with HTTP code and latency. Up to 4 probes run in parallel. See
[`scripts/osint-status.sh`](https://github.com/Opita-Code/dark-research-mcp/blob/main/scripts/osint-status.sh)
for the full list of URLs and the classification rules.

A backend returning `auth_required` (401/403) is *not* counted as a
failure: it means the endpoint exists but you would need an API key
to use it. A `timeout` or `fail` is a real signal that the backend
is down or has changed its API in a way our fixtures no longer
match.
