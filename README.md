## ВҝQuГ© hace?

**dark-research-mcp** es un servidor MCP escrito en Go que entrega a tu agente IA **57 herramientas especializadas** agrupadas en tres oficios:

1. **рҹ”Қ InvestigaciГіn (OSINT)** вҖ” 15 herramientas que enrutan consultas a backends nicho (OSV.dev, OpenAlex, RIPE, crt.sh, abuse.ch, DuckDuckGo, GDELT, Wayback, Ahmia, HIBP, ip-api, GitHub, crates.io, npm) con fallback automГЎtico.
2. **рҹҢҠ Vibe-flow** вҖ” 22 herramientas para gestionar el ciclo completo de producciГіn asistida por IA: spec (create/update/delete/render) вҶ’ artifact (log/update/delete) вҶ’ drift вҶ’ reconcile вҶ’ publish, con brand y compliance como reference data.
3. **вҡ–пёҸ Dark-ssd (LLM-as-judge)** вҖ” 8 jueces LLM: brand fit, compliance jurisdiccional, drift spec-vs-artifact, grounding de claims OSINT, **PII detection (GDPR/CCPA)**, **prompt-injection scan** (security gate antes de pasar texto no confiable al agente), y **consensus** (multi-sample judging para verdicts de alto riesgo).

Una sola base SQLite (`dark.db`) compartida con `dark-eval`. Una sola API. Un solo binario (~17 MB). **Sin magia: con cГіdigo que puedes leer y modificar.**

> рҹҮЁрҹҮҙ *Construido en Colombia como parte del ecosistema [Opita Code](https://opitacode.com). Software prГЎctico para investigaciГіn real, no para verse bonito en una presentaciГіn.*

## Papers y mentalidad

Este proyecto se apoya en cuatro lГӯneas de pensamiento explГӯcitas:

1. **SSOT (Single Source of Truth)** вҖ” un solo `dark.db`, una sola API, una sola versiГіn de la verdad. Si ves datos contradictorios en algГәn lado, el bug estГЎ en la consulta, no en el modelo.
2. **Closed-loop validation** вҖ” cada output creativo pasa por LLM-as-judge + persistence + reconcile. Sin esto, vibe-coding es un casino.
3. **Token economy (Atlan 2026)** вҖ” el cache LLM + los list endpoints filtrables son el primer paso para que `dark-ssd` no te queme el presupuesto en audit. MГЎs detalles en el skill `vibe-flow`.
4. **Open-spec, open-source, open-data** вҖ” todo el cГіdigo es legible. Las versiones de prompts se persisten en `sdd_evaluations.prompt_version` para que puedas reproducir cada verdict.

### Backend status monitoring

Los 16 backends OSINT no son nuestro cГіdigo вҖ” son servicios externos
que pueden cambiar de API o caerse. Por eso tenemos dos capas de
observabilidad, no solo una:

1. **VCR fixtures** (`internal/research/testutil/` + `fixtures/`) вҖ” los
   tests del parser y del router corren contra respuestas grabadas
   reales. CI no depende de la red, y un cambio de formato en un
   backend falla el test en segundos, no en producciГіn.
2. **Live status probe** (`scripts/osint-status.sh`) вҖ” corre en cada PR
   (`osint-smoke` job) y semanalmente (`osint-status` workflow).
   Sondea los 16 backends en paralelo, escribe
   [`BACKEND_STATUS.md`](BACKEND_STATUS.md) con HTTP code + latencia
   por backend, y auto-commit'a los cambios. Outages aparecen visibles
   sin abrir un issue.

Para refrescar fixtures cuando un backend cambia su API:

```sh
RECORD_FIXTURES=1 go test -count=1 -run 'TestRouter_' ./internal/research/
git diff fixtures/   # revisar el diff antes de commit
```
